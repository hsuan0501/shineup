# 🎯 任務系統重新設計 - 簡報內容

## 📋 背景與目的

### 初始問題
- **課程限制**：4個月的全棧開發專題課程
- **功能複雜性**：原任務描述涉及複雜外部API集成（政府資料庫、央行API、台灣證交所API）
- **實施可行性**：超出課程時限內的實現能力

### 解決方案
將 30 項任務重新設計為：**描述真實、可實現的後端功能** ← 而非虛擬或過度理想化的功能

---

## 📊 任務重設計概覽

### 整體框架
- **任務總數**：30 項（保持不變）
- **分類**：5 大類別，各 6 項任務
- **積分體系**：保留原有的升級積分與兌換積分機制
- **使用者等級**：Explorer → Creator → Visionary → Luminary

---

## 🔄 設計變化對比

### ❌ 舊設計問題
```
任務描述："調用政府資料庫驗證身份"、"調用銀行API"、"調用台灣證交所API"
實際困境：需要真實的政府/銀行API授權，4個月內無法實現
```

### ✅ 新設計方向
```
任務描述：明確指出後端資料庫表、數據驗證邏輯、簡單的API端點
實際優勢：可在課程期間完整實現，對應真實的全棧開發技能
```

## 🗄️ 核心資料庫設計概念

任務重設計引導實現以下資料庫表結構：

```
基礎用戶表
├── users (基本資訊)
├── user_profiles (KYC驗證、風險偏好)
├── user_accounts (銀行帳戶)
├── user_levels (等級進度)
└── user_achievements (成就記錄)

數據記錄表
├── activity_logs (登入記錄)
├── financial_goals (理財目標)
├── investment_portfolios (投資組合)
├── investment_plans (定投計畫)
├── trading_records (交易記錄)
├── performance_history (績效歷史)
└── risk_profiles (風險評估)

社群與互動表
├── referral_relationships (推薦關係)
├── forum_posts (論壇貼文)
├── forum_comments (論壇評論)
├── donations (捐款記錄)
└── contribution_scores (貢獻值統計)

配置與規則表
├── alert_rules (提醒規則)
├── risk_control_rules (風控規則)
├── audit_logs (審計日誌)
└── system_notifications (通知記錄)
```

## 💡 設計亮點

### 1. **與後端課程目標對齊**
- ✅ 資料庫 CRUD 操作
- ✅ API 端點設計
- ✅ 資料驗證與加密
- ✅ 複雜業務邏輯計算
- ✅ 統計聚合查詢
- ❌ 外部API整合（排除）

### 2. **循序漸進的難度**
- **日常任務**：簡單的CRUD與狀態管理
- **學習任務**：進度追蹤與計算邏輯
- **投資任務**：複雜的交易系統與關聯
- **永續任務**：標籤與社群功能
- **成就任務**：綜合的業務邏輯與統計

### 3. **實際工作場景還原**
- 每個任務代表真實的金融應用功能
- 後端描述明確指出需要實現的資料表和邏輯
- 避免過度承諾無法實現的功能

### 4. **前後端配合完整**
- 前端：任務卡片展示、模態框詳情、積分動畫
- 後端：資料存儲、驗證、計算、統計
- 全棧：API端點、狀態管理、使用者互動流程
